# Knowledge-vault schema

This vault stores lessons-learned, decisions, and reference notes
captured by AI agents (Claude Code, Codex) and humans editing in
Obsidian (desktop via LiveSync, browser via the cluster-hosted
instance). Notes are atomic — one concept per file — and form a
typed graph via `[[wikilinks]]` and `kb_relations` rows in the
knowledge-api Postgres.

## Top-level layout

```
topics/<topic-slug>/<kebab-title>.md       # cross-project: languages, frameworks, tools
projects/<github-repo-name>/<kebab-title>.md  # repo-specific
agents/_shared/<kebab-title>.md            # visible to every assistant
agents/<agent-name>/<kebab-title>.md       # one assistant only
_inbox/<YYYY-MM-DD>/<HHMMSS>-<slug>--<id8>.md  # untriaged captures (worker writes here)
_inbox/_needs-review/<slug>--<id8>.md      # curator flagged for human resolution
_index/recent.md                           # auto-generated by curator
_index/conflicts.md                        # auto-generated by curator
_index/topics/<topic-slug>.md              # MoC per topic, auto-generated
```

No top-level `notes/`, `personal/`, `work/`, or `public/` directories.
A note's primary home is one folder; cross-references happen through
`[[wikilinks]]` and `see_also` frontmatter, NOT by duplicating files.

### Slug rules

- Filename: `kebab-case-title.md`, lowercased, non-alphanumerics
  collapse to `-`, truncated at a word boundary to ≤60 chars.
- Filename collisions add `-2`, `-3` suffixes — newest collision
  wins the lower number.
- The canonical identity is the ULID in frontmatter, not the
  filename. Renames are free; the id never changes.

### Topic vocabulary

`<topic-slug>` is drawn from a closed list seeded in
`platform/cluster/flux/apps/knowledge/knowledge-curator/topics.yaml`
in the personal-stack monorepo. Adding a new topic requires a
deliberate PR — keeps the directory shape stable and prevents
synonym drift (`kotlin`, `Kotlin`, `kt`). The curator agent rejects
captures with unknown topic slugs and queues them under
`_inbox/_needs-review/` for human resolution.

## Frontmatter

Every note's first block:

```yaml
---
id: 01HXYZ00000000000000000000 # ULID, assigned by knowledge-api
type: lesson | decision | note | fact
scope: topic:<slug> | project:<repo> | agent:<name> | agent:_shared | _inbox
source: claude-code | codex | manual | url:<…>
captured_at: 2026-05-18T11:24:32+00:00
session_id: <claude-session-id> # optional
confidence: 0.4 # 0.0–1.0
supersedes: [01HABC…] # optional; demotes the named notes
see_also: [01HDEF…] # optional; symmetric link
tags: [k3s, postgres] # free-form, but prefer the topic folder
---
# Title

Body in markdown. Use [[Other Note Title]] or [[01HABC…]] to link
by title or by id. Atomic notes target 200–600 words; longer
narratives belong in `_index/`-style MoCs.
```

| Field         | Required | Notes                                                                  |
| ------------- | -------- | ---------------------------------------------------------------------- |
| `id`          | yes      | ULID. Stable across renames. Never edit by hand.                       |
| `type`        | yes      | Drives retrieval weight: lessons + decisions outrank notes + facts.    |
| `scope`       | yes      | Folder-derived primary scope. `_inbox` for untriaged.                  |
| `source`      | yes      | Provenance. `url:<…>` for ingested external pages.                     |
| `captured_at` | yes      | ISO 8601 UTC. Set at capture; never updated.                           |
| `session_id`  | no       | Originating chat. Lets `recall` link a note back to its session.       |
| `confidence`  | yes      | Worker starts at 0.4. Curator raises to 0.7-0.9 on consolidation.      |
| `supersedes`  | no       | Each named id has its `confidence` multiplied by 0.2 (kept for audit). |
| `see_also`    | no       | Symmetric. Curator can write these automatically over a threshold.     |
| `tags`        | no       | Free-form; sparse. Use the topic folder as the primary classification. |

### Closed enums

- `type` ∈ `lesson | decision | note | fact`.
- `scope` matches one of:
  - `topic:<topic-slug>` (slug from `topics.yaml`)
  - `project:<github-repo-name>`
  - `agent:<name>` or `agent:_shared`
  - `_inbox` (sentinel — curator-classified or pending classification)
- `source` is free-form but conventionally one of the values above.

## Conflict handling

A new note that conflicts with an existing one must either:

- include the prior note's id in `supersedes`, OR
- get a `kb_relations` row with `predicate=contradicts` between them.

Unhandled conflicts (curator confidence too low to auto-merge, or two
notes scored as overlapping above the dedup threshold) land in
`_index/conflicts.md` for human resolution.

## Three-writer rule

Three independent processes touch this git tree:

1. **knowledge-ingest-worker** — writes to `_inbox/` ONLY. Never
   touches `topics/`, `projects/`, `agents/`, or `_index/`.
2. **knowledge-curator** — reads `_inbox/`; `git mv`s files into the
   final folder; rewrites frontmatter; regenerates `_index/`. Writes
   to anywhere outside `_inbox/<day>/` raw captures.
3. **Humans via Obsidian** — write anywhere. The curator skips
   hand-edited notes that already carry a non-inbox `scope`.

To keep this safe:

- `.gitattributes` sets `merge=union` on `_index/**` so curator
  rewrites never produce merge conflicts.
- Curator pulls + rebases before every write, pushes immediately,
  and uses `git mv` so LiveSync's CouchDB-backed Obsidian sessions
  see clean renames.

## Indexes

`_index/` is curator-owned. Hand edits there get overwritten on the
next curator pass. Browse the folder tree (or use Obsidian Dataview)
for views the curator does not generate.

- `_index/recent.md` — 50 most-recently-promoted notes, newest first.
- `_index/conflicts.md` — unresolved supersedes / contradicts pairs.
- `_index/topics/<topic-slug>.md` — Map of Content per topic.
